## 数据并行
---
数据并行（Data Parallelism）是把同一个模型副本放在多张 GPU 上，让每张卡处理不同的数据分片，然后把各卡梯度聚合后再同步回去更新参数。核心目标是：在不改模型结构的前提下，用更多设备提升吞吐。
### DP
![[file-20260306143250969.png]]
DP（经典数据并行）通常基于参数服务器（Parameter Server）思路：Worker 负责计算，Server 负责聚合梯度（或同时更新参数）。

核心流程：
1. 每个 Worker（GPU）保存一份完整模型参数。
2. 一个 batch 被切分给不同 Worker，各自做前向/反向得到本地梯度。
3. Worker 将梯度 `push` 到 Server 做聚合（一般是累加）。
4. Worker 再从 Server `pull` 聚合后的梯度（或参数）并保持副本一致。

特点与问题：
- 架构直观、实现简单，适合单机多卡或小规模训练。
- 显存冗余高：每张卡都保存完整模型。
- 通信压力集中：Server 与所有 Worker 通信，跨机时容易被带宽限制。

异步梯度更新：
- 为减少等待，Worker 可在未拿到最新聚合结果时先继续下一轮计算。
- 好处是提高设备利用率；代价是梯度延迟（staleness）可能降低收敛速度或稳定性。

### DDP

DDP（分布式数据并行）是更主流的数据并行方案，常用于多机多卡。它通过去中心化通信避免单点 Server 瓶颈。

典型同步方式是 Ring-AllReduce，由两步组成：
- **Reduce-Scatter**：相邻 GPU 交换并累加梯度分块，最后每张卡得到一块全局聚合后的分片。
	
	Reduce-Scatter可以用一下图例来理解，假设有4块GPU，每个GPU上的数据被分成4块，比如GPU0的数据是 $a_0, b_0, c_0, d_0$。 Reduce-Scatter步骤每块GPU都只和相邻的2块GPU做通信（数据传递），并将接收到的数据块与自己本身持有的相同位置的数据块进行累加。

---
有一个可能的疑问就是，为什么要对GPU上的梯度切分成四份，再做ReduceScatter和AllReduce？可以不切分么？切分数一定要和卡数相等么
答：
可以不切分么？不可以！
假设你有**8 张 GPU 卡**，模型有一个 10 亿参数的梯度张量（约 4GB，float32）：
 1. 降低单次通信的数据量（最核心）

	- **不切分（纯 allreduce）**：每张卡要把 4GB 的完整梯度发给所有卡，全网总通信量是 `8卡 × 4GB × (8-1) = 224GB`（环形通信算法）。
    
	- **切分成 8 份（reduce_scatter）**：每张卡只处理 4GB/8=0.5GB 的分片，全网总通信量是 `8卡 × 0.5GB × (8-1) = 28GB`（仅为原来的 1/8）。
	
	通信是分布式训练的瓶颈，切分能**线性降低通信量**，这是最关键的收益。
2. 降低单卡显存峰值
	切分后，每张卡在 reduce_scatter 阶段只需要存储 “自己负责的分片梯度”，而不是完整梯度。比如上述例子，单卡显存占用从 4GB 降到 0.5GB，对大模型训练（动辄上百 GB 梯度）至关重要。
3. 通信与计算重叠（工程优化）
	切分后，每张卡可以在处理自己分片的同时，让通信库（如 NCCL）异步处理分片传输，实现通信和计算的重叠，进一步提升训练效率。

	结论：
	- DDP 与传统 DP 在总通信量是相同的，但通信负载更均衡。
	- 在多机场景下，DDP 通常有更好的扩展性与实际训练效率。
切分数一定要和卡数相等么？切分树和卡数相等是最优
一共有三种情况：
 1. 切分数 = 卡数（最优）
	 这是 DDP 的标准实现（如 PyTorch DDP 默认），因为：
	- 每张卡刚好负责 1 份分片，负载完全均衡；
	- 通信量降到最低（分片大小 = 总梯度 / 卡数）；
	- NCCL 等通信库对 “切分数 = 卡数” 做了深度优化，性能最优。
2. 切分数 < 卡数（比如 8 卡切 4 份）
	- 结果：每张卡要负责多个分片，负载不均衡（部分卡处理更多数据），通信量不会进一步降低，反而可能因负载不均导致整体变慢。
	- 适用场景：极特殊的硬件拓扑（比如多机多卡，同机内卡通信更快，可按 “机” 切分）
3. 切分数 > 卡数（比如 8 卡切 16 份）
	- 结果：每张卡负责多个分片，通信量不变（总分片数 / 卡数 = 等效切分数），但会增加分片 / 拼接的计算开销，得不偿失。
	- 适用场景：几乎没有，仅理论研究中可能出现。
