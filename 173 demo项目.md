## 我建议它至少体现 5 个点

### 1. 多轮状态管理

不是用户问一句答一句，而是能持续维护：

- 求职目标
- 目标城市/岗位
- 已投岗位
- 面试反馈
- 简历版本
- 待补项目

这能体现你对 agent state / memory 的理解，而记忆与持久执行本身也是主流 agent 框架强调的能力。

### 2. 多工具协同/多Skills

比如：

- JD 解析
- 简历解析
- 项目经历重写
- 岗位匹配评分
- 面试问题生成
- 投递计划生成

这样才能体现 Tool Use 、workflow和Skills，而不是单纯聊天。

### 3. 知识增强

至少接入一个小型知识库：

- 岗位 JD 库
- 你的项目经历库
- 面试题库
- 公司资料库

并且输出时给出依据或引用，而不是只给结论。  
这会明显提升“可信感”。

### 4. 评测 / 观测

哪怕是简单版，也建议做：

- trace 展示
- tool 调用日志
- 失败 case
- 一个小的离线评测集

现在 agent 项目越来越强调 observability 和 evaluation，这会让你的项目从“demo”变成“认真做过工程”。

### 5. MCP 或标准化工具接口

MCP 现在已经是很有辨识度的加分项，而且官方规范和 registry 都在快速发展。  
你的项目如果能写出：

- 支持 MCP tool
- 预留 MCP server/client 接口
- 或至少有 MCP-ready 的架构

会很加分。

# 对你的最佳策略

我建议你不要把题目定义成“求职助手 Agent”，而是把它包装成更强一点的名字，比如：

**Job Search Copilot / Career Agent / 求职工作流 Agent**

因为“助手”听起来容易像泛功能聊天机器人，  
而 “Copilot / Workflow Agent” 更像一个真正的系统。

# 一句话建议

**求职助手 Agent 可以做，但建议升级成“求职工作流 Agent / Job Search Copilot”，把重点放在多轮状态、知识增强、工具协同、证据输出和评测观测上。**  
这样它才配得上放在 GitHub 里当展示项目。
