# Windows 使用 Docker 部署 Milvus 教程

本文整理了在 **Windows** 环境下，使用 **Docker Desktop** 部署 **Milvus Standalone** 的完整流程，并补充说明了很多视频教程里会提到的一个关键步骤：

> 先在 `D:\milvus` 目录下新建 6 个空文件夹，再执行 `docker compose up -d`

这一步并不是“Milvus 固定要求必须建 6 个目录”，而通常是因为教程使用了 **宿主机目录挂载（bind mount）** 的 `docker-compose.yml`，需要提前准备本地数据目录，用于 **持久化、排错和配置映射**。Milvus 官方的 Standalone 部署本身依赖 3 个核心服务：`etcd`、`minio`、`milvus-standalone`。

---

## 1. 部署原理简介

Milvus Standalone 一般会通过 Docker Compose 启动以下 3 个核心容器：

- **etcd**：存储元数据
- **minio**：对象存储
- **milvus-standalone**：Milvus 主服务

官方 FAQ 也说明，Standalone 正常启动时，通常至少会看到这 3 个核心容器处于运行状态。

---

## 2. 环境要求

建议准备以下环境：

- Windows 10 / 11
- 已启用 **WSL 2**
- 已安装 **Docker Desktop**
- Docker Desktop 已正常启动
- PowerShell 或 Windows Terminal 可正常使用

先执行下面命令确认 Docker 可用：

```powershell
docker --version
docker compose version
```

如果都能返回版本号，说明环境已经基本就绪。

Milvus 官方对 Windows Docker 部署的说明中，也推荐使用 Docker Desktop，并指出在 Windows + WSL2 场景下，若使用目录挂载，通常将数据放在 Linux 文件系统中会更稳、更快。

---

## 3. 创建工作目录

先新建一个专门用于部署 Milvus 的目录：
```shell
mkdir D:\milvus  
cd D:\milvus
```

---
## 4. 为什么很多教程要先新建 6 个空文件夹

很多视频教程会要求你先在 `D:\milvus` 下新建 6 个空文件夹，然后再执行 `docker compose up -d`。

### 原因一：宿主机目录挂载需要本地目录存在

如果 `docker-compose.yml` 里使用的是类似下面这种写法：

volumes:  
  - D:/milvus/etcd:/etcd  
  - D:/milvus/minio:/minio_data  
  - D:/milvus/milvus:/var/lib/milvus

那么左边的 `D:/milvus/...` 就是 **Windows 本地目录**。这些目录通常用于挂载：

- etcd 数据
- MinIO 数据
- Milvus 数据
- 配置文件目录
- 日志目录
- 其他运行目录

提前建好这些目录，可以避免 Docker 在 Windows 下自动建目录时出现权限、共享盘、路径识别异常等问题。

### 原因二：实现数据持久化

如果不做本地挂载，很多数据默认只存在容器内部。容器删除后，这些数据可能也会一起消失。

提前创建本地目录并挂载后，有这些好处：

- 重启容器后数据还在
- 删除容器后数据还在
- 方便备份和迁移
- 方便排查问题

### 原因三：方便查看日志和配置

把日志目录、配置目录映射到宿主机后，你可以直接在 `D:\milvus` 下面查看数据和配置，而不必每次都进入容器内部。

### 原因四：这只是该教程对应 compose 文件的设计方式

需要注意：

> **不是 Milvus 固定要求必须创建 6 个目录**  
> 而是因为该视频使用的 `docker-compose.yml` 把不同数据目录拆分出来了。

也就是说：

- 有的 compose 只挂载 3 个目录
- 有的 compose 会挂载 6 个目录
- 有的 compose 不使用本地目录，而是使用 Docker named volumes

所以“先建 6 个目录”本质上只是为了配合那个教程里的 compose 文件。Milvus 官方提供的 compose 部署方式，也是围绕容器和卷挂载来完成的。

---

## 5. 推荐先创建的目录

如果你想按教程思路操作，可以在 `D:\milvus` 下先手动创建几个目录。名称不一定完全固定，但常见会包括：

- `conf`
- `logs`
- `db`
- `wal`
- `volumes`
- `pic`

在 PowerShell 中可以这样创建：

mkdir D:\milvus\conf  
mkdir D:\milvus\logs  
mkdir D:\milvus\data  
mkdir D:\milvus\wal  
mkdir D:\milvus\etcd  
mkdir D:\milvus\minio

> 说明：具体目录名仍然要以你使用的 `docker-compose.yml` 为准。  
> 如果 compose 文件里写的是别的名字，就按 compose 文件里的路径创建。

---

## 6. 获取 docker-compose.yml

Milvus 官方提供了 Standalone 的 Docker Compose 文件，可直接下载后使用。官方文档示例中提供了对应版本的 `milvus-standalone-docker-compose.yml`。

例如在 `D:\milvus` 目录下执行：

Invoke-WebRequest -Uri "https://github.com/milvus-io/milvus/releases/download/v2.6.13/milvus-standalone-docker-compose.yml" -OutFile "docker-compose.yml"

如果你用的是视频作者提供的 compose 文件，则把它放到 `D:\milvus` 目录即可。

---

## 7. 一个常见的目录挂载思路示例

很多教程的核心其实就是把容器中的关键目录映射到 Windows 本地目录。

例如：

services:  
  etcd:  
    volumes:  
      - D:/milvus/etcd:/etcd  
  
  minio:  
    volumes:  
      - D:/milvus/minio:/minio_data  
  
  standalone:  
    volumes:  
      - D:/milvus/data:/var/lib/milvus  
      - D:/milvus/conf:/milvus/configs  
      - D:/milvus/logs:/var/log/milvus  
      - D:/milvus/wal:/milvus/wal

上面只是示意，不同版本的 compose 文件路径可能不同。  
你在实际使用时，应以手头的 `docker-compose.yml` 为准。

---

## 8. 启动 Milvus

在 `docker-compose.yml` 所在目录执行：

docker compose up -d

参数说明：

- `up`：启动服务
- `-d`：后台运行

---

## 9. 查看容器是否启动成功

执行：

docker ps

如果正常启动，你通常会看到类似以下容器：

- `milvus-etcd`
- `milvus-minio`
- `milvus-standalone`

Milvus 官方 FAQ 说明，Standalone 正常运行时一般至少会看到这 3 个核心容器。

---

## 10. 查看日志

如果启动失败或连接异常，可以先查看日志。

查看所有服务日志：

docker compose logs

查看 Milvus 主服务日志：

docker logs milvus-standalone

如果容器名不同，可以先执行：

docker ps

确认实际名称。

---

## 11. 检查端口

Milvus 常用端口通常包括：

- `19530`：客户端连接端口
- `9091`：管理或健康检查相关端口

执行下面命令可以查看端口映射：

docker ps

如果映射正常，客户端一般可以通过下面地址连接：

- Host: `127.0.0.1`
- Port: `19530`

---

## 12. 使用 Python 测试连接

先安装 Python 客户端：

pip install pymilvus

新建 `test_milvus.py`：

from pymilvus import connections  
  
connections.connect(  
    alias="default",  
    host="127.0.0.1",  
    port="19530"  
)  
  
print("Milvus 连接成功")

运行：

python test_milvus.py

如果输出：

Milvus 连接成功

说明服务已经部署完成。

---

## 13. 停止与重启

停止服务：

docker compose down

重新启动：

docker compose up -d

重启某个容器：

docker restart milvus-standalone

---

## 14. 常见问题

### 14.1 Docker Desktop 没有启动

表现：

- `docker compose up -d` 报错
- 无法连接 Docker daemon

解决方法：

- 启动 Docker Desktop
- 等待 Docker Engine 完全可用
- 再重新执行命令

---

### 14.2 端口被占用

检查端口占用：

netstat -ano | findstr 19530

如果被占用，可以修改 `docker-compose.yml` 的端口映射，例如：

ports:  
  - "19531:19530"

然后重新启动：

docker compose down  
docker compose up -d

连接时端口改为 `19531`。

---

### 14.3 拉取镜像失败

可以先单独拉取镜像：

docker compose pull  
docker compose up -d

如果网络较慢，可以检查 Docker 镜像加速配置。

---

### 14.4 容器在运行，但客户端连接失败

建议按顺序检查：

1. `docker ps` 看 3 个核心容器是否都在运行
2. `docker compose logs` 看日志是否报错
3. 确认 `19530` 已正确映射
4. 确认防火墙未拦截
5. 确认连接地址是 `127.0.0.1:19530`

---

### 14.5 为什么已经创建目录了，还是报挂载问题

可能原因包括：

- Docker Desktop 没有共享对应磁盘
- Windows 路径写法不对
- compose 文件里的目录名和你创建的不一致
- 权限不足
- WSL2 / Docker Desktop 状态异常

---

## 15. 关于 Windows 挂载路径的补充说明

虽然很多教程会直接使用 `D:\milvus` 这样的 Windows 路径挂载，但 Milvus 官方建议，在 Windows + WSL2 场景下，若使用 bind mount，优先考虑将数据存放在 Linux 文件系统中，以获得更好的兼容性和性能。也就是说：

- **教程写法更直观，适合入门**
- **官方建议的 Linux 文件系统挂载更稳一些**

这两种思路并不冲突。

## 17. 总结

在 Windows 上通过 Docker 部署 Milvus，最常见的流程就是：

1. 安装 Docker Desktop
2. 启用 WSL 2
3. 创建 `D:\milvus` 工作目录
4. 根据 `docker-compose.yml` 提前创建本地挂载目录
5. 执行 `docker compose up -d`
6. 用 `docker ps` 和客户端连接验证服务

而“先创建 6 个空文件夹”的原因，本质上是：

> **为了配合 compose 文件中的宿主机目录挂载，实现持久化、便于排错，并减少 Windows 路径和权限问题。**

它不是 Milvus 强制要求的固定步骤，而是很多教程为了更稳妥、更直观，采用的一种部署方式。
