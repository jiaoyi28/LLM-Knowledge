# RAG

RRF进行多路检索（向量、文本、图）的排序融合

## 索引召回
### 倒排索引
> 倒排索引：记录每个词，在哪些文档中出现。使用于召回阶段

比如有三篇文档：
- D1: `我喜欢机器学习`
- D2: `机器学习很有趣`
- D3: `我喜欢搜索引擎`

分词后，倒排索引可能长这样：
- 我 → D1, D3
- 喜欢 → D1, D3
- 机器学习 → D1, D2
- 很有趣 → D2
- 搜索引擎 → D3

>作用是通过关键词，快速找到相关文档集

## 排序
### BM25
> BM25：用来衡量查询词和文档的相关性大小。使用于排序阶段

它主要考虑几个因素：
1. 词频（TF）：一个词在某篇文档里出现越多，通常说明这篇文档和这个词越相关。
2. 逆文档频率（IDF）：如果一个词几乎所有文档里都有，比如“的”“是”，那它区分文档的能力就很弱，权重应该低。  如果一个词只出现在少数文档里，那它更有区分力，权重更高。
3. 文档长度归一化：长文天然更容易包含更多词，不能因为它长就占便宜，所以 BM25 会对文档长度做调整。

你不一定要记公式，但它大致是在做：

> **相关性 = 查询词的重要程度 × 在该文档中的匹配强度 × 长度修正**

## 多路排序融合
通常采用多路检索系统，比如向量、关键词、图，的时候，每一路都有自己的排序，需要进行排序统合
### RRF
> RRF（倒数排序融合）：通过计算文档在不同索引中的倒数排名分数来提升综合相关性。

RRF 的公式很简单：$$RRF(d) = \sum_i \frac{1}{k + \text{rank}_i(d)}$$

其中：
- $d$ 是某个文档
- $i$ 表示第$i$个排序器 / 检索器
- $\text{rank}_i(d)$ 是文档 $d$ 在第 $i$ 个结果列表中的排名
- $k$ 是一个平滑常数，常见取值比如 **60**

意思就是：
- 排名越靠前，贡献越大
- 排名越靠后，贡献越小
- 如果一个文档在多个列表里都排得不错，它最终总分就会高

# MCP

## 基本概念
**MCP** 全称 **Model Context Protocol**，可以理解成给大模型用的“通用外设协议”。

它解决的问题是：  
**让模型用统一方式连接外部工具、数据和服务**，而不是每接一个工具都写一套专用接口。

你可以把它类比成：

- **USB** 对电脑：统一接键盘、鼠标、硬盘
- **MCP** 对模型：统一接搜索、数据库、文档库、代码工具、业务系统

核心里只有三件事：

**1. Host**  
运行模型的应用，比如 Cursor、Claude Desktop、你自己的 Agent。

**2. Client**  
Host 里负责和 MCP server 通信的那层。

**3. MCP Server**  
把某种能力暴露给模型，比如：

- 查文档
- 读数据库
- 调接口
- 搜代码
- 操作本地文件

模型本身不直接懂这些外部系统，**它是通过 MCP server 间接使用能力**。
## 部署与transport
MCP server常见的是两种部署形态
### 1. 本地启动型

这是很多开发者最常见的方式。  
MCP 最早很常见的是 **`stdio`** 这种 transport：你的客户端启动一个本地进程，通过标准输入输出跟它通信。LangChain 的 MCP 文档也还在大量展示这种用法。

适合：

- 本地开发
- 个人工具
- 跟本机文件、IDE、终端强绑定的工具

特点：

- 不一定有公网地址
- 往往需要你“运行一个 server 进程”
- 不是“部署到云上给大家访问”的思路

### 2. 远程 URL 型

现在官方生态已经明确支持 **remote MCP server**。  
MCP 官方文档把 **Streamable HTTP** 作为远程 MCP 的 transport；OpenAI 官方工具文档也直接写了，remote MCP servers 可以是公网上实现了 MCP 的服务器。

适合：

- 团队共享工具
- SaaS/平台能力
- 需要 OAuth、API key、统一鉴权
- 服务端 Agent 调用

特点：

- 通常是一个 URL
- 不需要你在每台客户端机器上单独起进程
- 更适合生产环境和多用户场景 

## transport
> 是不是这两种类型分别对应stdio transport和http transport？transport都有哪些

基本可以这样理解，但**不是完全一一对应**。

### 先说结论

- **本地型 MCP**：通常对应 **`stdio`**
- **远程 URL 型 MCP**：通常对应 **HTTP 类 transport**
- 但 **HTTP 不一定等于“公网第三方提供”**，也可以是你自己本地起一个 `http://localhost:3000/mcp` 服务
- 同样，远程场景主流是 HTTP，但某些 SDK/平台还会保留对旧式 SSE 兼容支持

### 当前标准 transport 主要有哪些

按 MCP 官方规范，**当前标准的 transport 主要是两个**：

1. **`stdio`**  
    通过标准输入/输出通信，最适合本地进程、本机工具、命令行集成。
2. **`Streamable HTTP`**  
    这是现在的 HTTP 方向标准 transport，适合本地或远程的 HTTP 服务。它可以用于 Web 集成、多客户端、会话管理这类场景。

### SSE 是什么关系

**HTTP+SSE（或叫 HTTP/SSE）是旧一些的 transport 形式，很多实现还兼容，但在 MCP 新规范里已经属于旧版兼容路径。**  
MCP 官方 2025-03-26 规范明确写了：客户端和服务端可以为旧版本协议保留对 **deprecated HTTP+SSE transport** 的兼容。OpenAI 的远程 MCP 文档也提到，Responses API 可接支持 **Streamable HTTP 或 HTTP/SSE** 的远程 MCP server。

所以你可以把它记成：

- **现在主推**：`stdio`、`Streamable HTTP`
- **历史兼容**：`HTTP/SSE` / `SSE`
## langraph使用案例
> 如果我想在我一个基于langraph开发的Agent中接入这个MCP服务，应该如何操作

**把 Context7 当成一个 MCP 工具源，先接到 `langchain-mcp-adapters` 的 `MultiServerMCPClient`，再把取到的 tools 挂到你的 LangGraph agent 上。**  
LangChain/LangGraph 官方文档明确支持这套方式；Context7 官方也明确提供了 MCP 模式，并且它的包支持 `stdio` 和 `http` 两种 transport。

我建议你优先这样选：

- **本地开发 / 单机 agent**：用 **`stdio`**，最简单最稳。
- **你要把 agent 跑在服务端**：优先考虑 **HTTP MCP**，因为 LangChain 官方文档特别提醒，`stdio` 更适合运行在用户本机的应用，而不是 Web server 场景。

## 方案一：最推荐，LangGraph 里直接接本地 `stdio` 版 Context7

### 1）安装依赖

```shell
pip install langgraph langchain langchain-openai langchain-mcp-adapters
```

你本机还需要能跑 Node/NPM，因为 Context7 官方最常见的安装方式是：

```shell
npx -y @upstash/context7-mcp
```

这是 Context7 官方 README 里的标准 MCP 接入方式之一。

---

### 2）在 Python 里把 Context7 注册成 MCP server
```python
import asyncio  
from langchain_openai import ChatOpenAI  
from langchain_mcp_adapters.client import MultiServerMCPClient  
from langgraph.graph import StateGraph, MessagesState, START, END  
from langgraph.prebuilt import ToolNode, tools_condition  
  
async def build_graph():  
    client = MultiServerMCPClient(  
        {  
            "context7": {  
                "transport": "stdio",  
                "command": "npx",  
                "args": ["-y", "@upstash/context7-mcp@latest"],  
            }  
        }  
    )  
  
    tools = await client.get_tools()  
  
    llm = ChatOpenAI(model="gpt-4.1")  
  
    def call_model(state: MessagesState):  
        system_hint = {  
            "role": "system",  
            "content": (  
                "你是一个代码助手。"  
                "当用户询问某个库、框架、SDK、API 的最新用法时，优先使用 Context7 工具。"  
                "先解析库 ID，再查询文档。"  
                "如果不是库/API问题，就正常回答。"  
            ),  
        }  
        msgs = [system_hint, *state["messages"]]  
        response = llm.bind_tools(tools).invoke(msgs)  
        return {"messages": [response]}  
  
    builder = StateGraph(MessagesState)  
    builder.add_node("call_model", call_model)  
    builder.add_node("tools", ToolNode(tools))  
  
    builder.add_edge(START, "call_model")  
    builder.add_conditional_edges("call_model", tools_condition)  
    builder.add_edge("tools", "call_model")  
  
    return builder.compile()  
  
async def main():  
    graph = await build_graph()  
    result = await graph.ainvoke(  
        {  
            "messages": [  
                {  
                    "role": "user",  
                    "content": "帮我写一个 Next.js 里间件校验 JWT 的例子，要求用最新写法。"  
                }  
            ]  
        }  
    )  
    print(result)  
  
if __name__ == "__main__":  
    asyncio.run(main())
```

这套写法是按 LangChain/LangGraph 官方 MCP 文档的模式来的：  
`MultiServerMCPClient(...) -> await client.get_tools() -> model.bind_tools(tools) -> ToolNode(tools)`。

## 方案二：如果你想让 LangGraph 服务端通过 HTTP 访问 Context7

Context7 官方 README 里写了它支持：

- `--transport <stdio|http>`
- `--port <number>`

例如可用 HTTP 跑在某个本地端口。

### 1）先启动 Context7 MCP 的 HTTP 服务

```shell
npx -y @upstash/context7-mcp@latest --transport http --port 3000
```

> 文档里展示的是源码运行方式 `node packages/mcp/dist/index.js --transport http --port 8080`；CLI 参数本身就是官方支持的。

### 2）LangGraph 用 HTTP transport 连接
```
import asyncio  
from langchain_openai import ChatOpenAI  
from langchain_mcp_adapters.client import MultiServerMCPClient  
from langgraph.graph import StateGraph, MessagesState, START  
from langgraph.prebuilt import ToolNode, tools_condition  
  
async def build_graph():  
    client = MultiServerMCPClient(  
        {  
            "context7": {  
                "transport": "http",  
                "url": "http://localhost:3000/mcp",  
            }  
        }  
    )  
  
    tools = await client.get_tools()  
    llm = ChatOpenAI(model="gpt-4.1")  
  
    def call_model(state: MessagesState):  
        response = llm.bind_tools(tools).invoke(state["messages"])  
        return {"messages": [response]}  
  
    builder = StateGraph(MessagesState)  
    builder.add_node("call_model", call_model)  
    builder.add_node("tools", ToolNode(tools))  
    builder.add_edge(START, "call_model")  
    builder.add_conditional_edges("call_model", tools_condition)  
    builder.add_edge("tools", "call_model")  
    return builder.compile()  
  
async def main():  
    graph = await build_graph()  
    result = await graph.ainvoke(  
        {"messages": [{"role": "user", "content": "查询 FastAPI 最新 lifespan 用法"}]}  
    )  
    print(result)  
  
if __name__ == "__main__":  
    asyncio.run(main())
```

LangChain MCP 适配器官方文档明确给出了 `transport: "http"`、`url: "http://localhost:3000/mcp"` 这种用法。

## Context7 里你实际会用到哪些工具

Context7 官方 README 里列出的核心工具是两个：

- `resolve-library-id`：先把“nextjs / react / fastapi”这种名字解析成 Context7 的库 ID
- `query-docs`：再按这个库 ID 去查相关文档片段

所以你可以在 system prompt 里明确要求模型按这个顺序用：

当需要查询库文档时：  
1. 先调用 resolve-library-id  
2. 再调用 query-docs  
3. 仅根据查到的文档回答

这样工具调用会更稳定。

## 一个更适合代码助手的 system prompt

你是一个面向工程开发的代码助手。  
  
规则：  
- 只要用户在问某个库、框架、SDK、API 的用法、版本差异、迁移方式、配置方式，优先使用 Context7。  
- 使用 Context7 时，先调用 resolve-library-id，再调用 query-docs。  
- 回答必须以查到的文档为依据，不要编造 API。  
- 如果用户问的是通用编程逻辑，不必强制调用 Context7。  
- 输出代码时，尽量给完整、可运行示例。

## 生产环境里的建议

### 1）不要把 Context7 当“所有问题都要查”的工具

它更适合这类问题：

- “React Router 最新写法”
- “Next.js App Router 中间件怎么做”
- “FastAPI lifespan 现在推荐怎么写”
- “Supabase auth 某个 API 怎么用”

对纯业务逻辑、数学、文本处理问题，没必要走 Context7。

### 2）服务端优先 HTTP，不优先 stdio

LangChain 官方提醒过，`stdio` 主要是给“运行在用户机器上的应用”设计的；如果你把 LangGraph 作为 API server 部署，最好评估 HTTP transport 或者干脆把必要能力改成普通 `@tool`。

### 3）把 Context7 做成“按需工具”

最常见做法是：

- 主 agent 正常推理
- 只有当用户问题里出现库名 / 框架名 / “最新版” / “官方文档” / “迁移” / “API” 之类信号时，再触发 MCP 工具

这样成本和延迟会更可控。

## 常见坑

### 1）`npx` 拉不起来

Context7 README 里给了几个排障建议：

- 改成 `bunx`
- 包名带 `@latest`
- 确保 Node.js v18+
- 某些环境下加 Node flags 处理 ESM / fetch / TLS 问题

例如：

{  
  "command": "bunx",  
  "args": ["-y", "@upstash/context7-mcp@latest"]  
}

### 2）工具拿到了，但模型不爱调用

通常是这几个原因：

- system prompt 没明确要求何时用 Context7
- 用户问题没提到具体库名
- 工具描述太泛，模型判断不出先后关系

解决办法就是在 prompt 里明确“先 resolve-library-id，再 query-docs”。

### 3）你以为可以直接连一个官方公网 MCP URL

截至我这次查到的公开资料，Context7 官方最明确、最文档化的方式仍然是运行 `@upstash/context7-mcp` 包；我没有看到一份足够明确的官方文档把一个稳定公网 MCP URL 作为主要接入方式公开出来。这个判断和前面一样，最好别把架构押在“官方一定提供固定公网 URL”上。

## 你可以直接照抄的最小可用版本

import asyncio  
from langchain_openai import ChatOpenAI  
from langchain_mcp_adapters.client import MultiServerMCPClient  
from langgraph.prebuilt import create_react_agent  
  
async def main():  
    client = MultiServerMCPClient(  
        {  
            "context7": {  
                "transport": "stdio",  
                "command": "npx",  
                "args": ["-y", "@upstash/context7-mcp@latest"],  
            }  
        }  
    )  
  
    tools = await client.get_tools()  
  
    agent = create_react_agent(  
        model=ChatOpenAI(model="gpt-4.1"),  
        tools=tools,  
        prompt=(  
            "你是代码助手。"  
            "当用户询问库、框架、SDK、API 的最新用法时，优先使用 Context7。"  
            "先调用 resolve-library-id，再调用 query-docs。"  
        ),  
    )  
  
    result = await agent.ainvoke(  
        {  
            "messages": [  
                {  
                    "role": "user",  
                    "content": "给我一个最新的 FastAPI lifespan 示例"  
                }  
            ]  
        }  
    )  
    print(result)  
  
asyncio.run(main())
