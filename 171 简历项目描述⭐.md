## 3 分钟版

### S（背景）

我有一个比较有代表性的项目是 Agent 应用开发，里面主要有两个部分，一个是手机图库 Agent，一个是类似 OpenClaw 的 Claw Agent。  
图库 Agent 的场景比较明确，就是让用户围绕手机中的图片，通过自然语言去做问答、搜索、理解相关的交互。这里面其实涉及多种能力，比如图片多模理解、图片检索、问答生成等。  
另外一个是 Claw Agent，它更像一个通用 Agent 应用体系，我在里面主要负责 Gateway 模块和 Skills 自演进。

### T（任务）

我在图库 Agent 里的核心任务，不是去做底层图片能力本身，而是把已有的图片能力真正组装成一个可运行的 Agent。因为图片多模理解和图片多路检索是其他同事提供的 Tool，所以我要解决的是：  
第一，用户的问题怎么理解；  
第二，当前问题应该选哪些 Tool；  
第三，不同 Tool 返回的信息怎么融合；  
第四，最后怎么生成一个合理的回答。  
也就是说，我负责的是 Agent 主流程和图片问答链路。

在 Claw Agent 里，我负责 Gateway 模块。这个模块本质上是 Agent 系统运行时的一个关键枢纽，要处理 cron、heartbeat、消息路由、channel 管理和 hook 这些事情。除此之外，我还做了一个 Skills 自演进的探索，希望让 Agent 能根据执行轨迹不断优化自己的 Skills。

### A（行动）

在图库 Agent 中，我先从场景出发设计了 Agent 的主执行链路。用户进来后，先做问题理解，再根据问题类型决定是否需要调用图片理解 Tool、图片检索 Tool，或者多种 Tool 组合调用。  
这里面的难点其实不在于单个 Tool，而在于 Tool 之间的协同。例如有些问题只需要检索，有些问题需要先检索再理解，还有些问题需要对多个结果做整合。  
所以我重点做了两件事：  
第一是工具编排，把 Tool 调用流程组织起来；  
第二是结果整合，把不同 Tool 的输出组织成最终回答。

在 Claw Agent 中，我主要实现了 Gateway 的几个核心能力：

- cron：支持定时任务触发
- heartbeat：支持 Agent 状态感知
- 消息路由：把请求分发到正确的处理链路
- channel 管理：管理不同通道下的会话和消息
- hook：支持链路中的扩展能力接入

这部分工作的重点是把 Agent 运行链路打通，让系统更稳定、更容易扩展。

在 Skills 自演进方面，我做的是基于 Agent 执行轨迹来优化 Skills。我的思路是，执行轨迹里其实包含很多有价值的信息，比如在哪一步调用错了 Tool、哪个 Skill 的触发条件不合理、某类任务总是在某一步失败。  
所以我尝试把这些轨迹信息转成对 Skills 的优化依据，让 Agent 的技能可以逐步改进。

### R（结果）

这个项目的结果，一方面是我真正做过 Agent 应用从“能力拼装”到“工程落地”的完整链路，而不只是某个算法模块。  
另一方面，我在里面形成了两类比较强的经验：  
第一类是应用层经验，比如 Agent 主流程设计、Tool Use 编排、结果融合；  
第二类是系统层经验，比如 Gateway、消息路由、状态管理、执行链路扩展。  
所以这个项目对我来说价值很大，它让我对 Agent 应用不仅有“会用”的理解，也有“怎么落地、怎么持续演进”的理解。

# 二、复杂知识检索 / Agentic Search / 图检索项目

---

## 1 分钟版

### S（背景）

我做过一个复杂知识检索增强项目，背景是部门在做自研的知识检索 RAG 引擎，原本已经有向量索引和倒排索引，但在复杂文档库场景下，面对关联性问题、归纳总结类问题时效果还不够理想。

### T（任务）

我的主要任务是负责 Agentic Search 和图检索方案的探索，目标是利用图结构增强复杂问题下的知识搜索和推理能力。

### A（行动）

我围绕图检索设计了搜索方案，把图结构引入检索链路，并重点探索了基于 Agentic Search 的搜索流程和 beam search 的路径搜索策略。核心想解决的是复杂问题下，如何更有效地发现多跳关系和更优搜索路径。

### R（结果）

最终在 HotpotQA 数据集上，相比 HippoRAG 大概有 5% 左右的效果提升。虽然这个方案最后没有正式业务落地，但完成了较完整的方案验证，也让我在 GraphRAG、复杂检索和 Agentic Search 方面积累了比较深入的经验。

---

## 3 分钟版

### S（背景）

我有一个技术深度比较强的项目，是复杂知识检索增强。背景是我们部门在做自研的知识检索 RAG 引擎，基础架构里已经有向量索引和倒排索引，但我们发现，在复杂文档库场景下，尤其是跨文档关联问题、需要多跳推理的问题，以及归纳总结类问题上，传统检索链路会有明显瓶颈。  
所以团队当时在探索引入图索引、GraphRAG 这类思路，而我重点负责的是其中 Agentic Search 和图检索这一部分。

### T（任务）

我的任务主要有两个：  
第一，设计一个适合图结构场景的搜索方案；  
第二，把 Agentic Search 的思路引进来，增强复杂问题下的知识定位能力。  
本质上要解决的问题是：面对一个复杂问题，系统怎么不只是“召回相关文档”，而是能沿着图中的关系去做更有效的搜索和推理。

### A（行动）

我在这个项目里主要做的是图检索和 Agentic Search 方案设计。  
一开始我们已经有了图索引这个基础，但如果只是把图建出来，价值还不够，关键是怎么在图上进行搜索。  
所以我主要从两个方向推进：

第一是 **Agentic Search**。  
我的理解是，复杂问题下的检索不应该只是一次性召回，而应该有点像 Agent 的搜索过程：

- 先根据当前问题定位一个搜索起点
- 再根据已有结果动态决定下一步往哪里扩展
- 最后形成一条更合理的知识搜索路径

第二是 **beam search 在图检索中的应用**。  
图搜索空间通常会比较大，如果不做约束，很容易走偏或者代价太高。所以我尝试用 beam search 来管理搜索过程中的候选路径，保留更有希望的搜索分支，减少无效扩展。  
这样做的核心目的是，在搜索效率和搜索质量之间找到一个平衡。

这个项目里的难点主要有两个：  
一个是复杂问题的搜索路径不明显，不像简单检索那样直接匹配关键词；  
另一个是图结构虽然能表达关系，但如果搜索策略不好，图也不一定真的带来收益。  
所以我做的重点不是“有图”，而是“怎么在图上搜索得更合理”。

### R（结果）

最终这个方案在 HotpotQA 数据集上，相比 HippoRAG 取得了大约 5% 的效果提升。  
虽然最后没有正式落到业务里，但我认为这个项目的价值还是很大的，因为它完成了从问题分析、方法设计、到实验验证的一整套闭环。  
对我个人来说，这个项目最大的收获是：  
第一，我对复杂知识检索、GraphRAG、图检索增强有了比较深入的理解；  
第二，我会从“搜索策略”的角度看检索，而不只是从“索引结构”的角度看检索；  
第三，我也积累了把 Agent 思路用于检索过程的实践经验。

## 3 分钟版

### S（背景）

我还做过一个比较有代表性的知识增强医疗 Agent 项目。这个项目的目标不是做一个完全自动诊断系统，而是做一个面向辅助初诊的 Agent：用户输入症状描述后，系统给出初步建议，比如可能相关的科室、医生推荐、相似病例，以及支撑这些结论的证据链。  
为了让这个 Agent 更可靠，我们整合了多种知识来源，包括开源医学知识图谱、开源病例库、PubMed 文献资源，以及网页搜索 Tool。

### T（任务）

我在这个项目里主要负责三块：  
第一，多轮对话 RAG 流程；  
第二，证据链生成；  
第三，PubMed MCP 接入。

我的核心任务其实是解决两个问题：  
一个是医疗咨询不是一次问答，经常是多轮补充描述，所以 RAG 流程要能处理上下文承接；  
另一个是医疗场景对可信性要求很高，所以系统不能只给结论，还要尽量给出回答依据。

### A（行动）

在多轮对话 RAG 流程上，我主要做的是把连续症状描述组织成一个可检索、可生成的上下文。  
因为医疗场景里，用户往往不会一次性把信息说全，可能先说“发烧、咳嗽”，后面再补充“持续三天”“夜间更严重”“有胸痛”之类的信息。  
所以如果每轮都单独处理，很容易丢失上下文，导致检索和回答偏掉。  
我做的事情就是把这些多轮信息更好地承接起来，形成更完整的问题表示，再去触发知识检索和生成。

在证据链生成上，我重点考虑的是怎么让结果更可解释。  
因为医疗场景里，如果 Agent 只给一个结论，没有依据，用户和医生都很难信任。  
所以我做的是把不同来源的证据组织起来，比如：

- 医学知识图谱里的结构化知识
- 相似病例中的症状和处理方式
- PubMed 文献中的相关内容  
    把这些信息整理成能支撑回答的依据链，而不是单纯给一个最终结论。

在 PubMed MCP 接入方面，我主要负责把外部文献检索能力接进整个 Agent 流程。  
这样 Agent 不只是依赖本地已有知识，还可以动态获取开放医学文献，提高知识覆盖范围和时效性。

### R（结果）

这个项目最大的结果，不一定是某个量化指标，而是让我形成了对高风险场景 Agent 的比较深的理解。  
第一，技术上我做过多轮对话 RAG、证据链生成和 MCP 接入这几个很核心的能力；  
第二，产品上我明显感受到，在医疗场景里，大家对 Agent 做“诊断”这件事会非常谨慎，所以系统设计必须更强调证据、可解释性和边界控制。  
这个经历让我对知识增强 Agent 不只是停留在“能不能答出来”，而是会更关注“为什么这么答、用户能不能信、场景能不能接受”。
